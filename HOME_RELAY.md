# Свой ноут как Middle-узел MurNet

Делаешь свой компьютер частью mesh-сети. Юридически безопасная роль — Middle не видит ни клиента, ни сайта.

## 🟢 Рекомендуемый способ — SSH reverse tunnel (без открытия портов)

**Работает у 100% юзеров**, включая CGNAT, мобильный интернет, корпоративные сети. Никаких настроек роутера.

```powershell
cd D:\kai\03_Projects\MURNET
.\home_node.ps1
```

Скрипт сам:
1. Стартует `home_relay.py` локально на 9292
2. Поднимает SSH-туннель к VDS: `VDS:9290 → твой_ноут:9292`
3. Автореконнект при разрыве

С этого момента ты — Middle-узел, доступный извне как `80.93.52.15:9290`. Трафик идёт через **исходящий** SSH к VDS, никаких входящих соединений твоему роутеру не приходит.

**Ctrl+C** в окне — узел отключается. Сеть автоматически перестроит circuits.

## 🔴 Альтернатива — прямое подключение с port-forwarding

Только если у тебя **публичный IPv4** (не CGNAT) и **админский доступ к роутеру**. Иначе используй SSH tunnel выше.

### Шаг 1 — открой порт на роутере (5 минут)

1. Зайди в админку роутера: `http://192.168.1.1` (или `192.168.0.1` / `192.168.1.1` / `tplinkwifi.net` — зависит от роутера)
2. Логин — обычно `admin / admin` или `admin / пароль с обратной стороны роутера`
3. Найди раздел **«Port Forwarding»** / **«Виртуальные серверы»** / **«NAT»**
4. Добавь правило:
   - **External port:** `9292`
   - **Internal port:** `9292`
   - **Internal IP:** твой локальный IP ноута (узнать: `ipconfig` в cmd → ищи `IPv4-адрес` под Wi-Fi или Ethernet, обычно `192.168.X.Y`)
   - **Protocol:** `TCP`
5. Сохрани, перезагрузи роутер если просит

**Проверка**: после открытия порта, на сайте https://2ip.ru/check-port → введи свой публичный IP и `9292` → должно быть «открыт». Если «закрыт» — провайдер использует **CGNAT**, и проброс не сработает (тогда нужен VDS вместо ноута).

## Шаг 2 — запусти узел

```powershell
cd D:\kai\03_Projects\MURNET
python home_relay.py
```

Скрипт:
- сам определит твой публичный IP
- начнёт слушать `0.0.0.0:9292`
- анонсирует себя VDS-Guard'у каждые 30 сек
- покажет статистику каждую минуту

## Шаг 3 — убедись что VDS тебя видит

В соседнем терминале:
```bash
ssh -i D:\kai\02_System\vds_key root@80.93.52.15 "tail -20 /var/log/murnet-hs.log"
```

Должна появиться строка типа:
```
[obfs] accept <твой публичный IP>:NNNNN
[стат] discovered relay <твой ноут>
```

## Шаг 4 — пустить circuit через свой ноут

Сейчас локальный `murnet_proxy.py` идёт через **VDS-Middle** (захардкожено). Чтобы переключить на свой ноут — открой `murnet_proxy.py:39` и замени:

```python
VDS_MIDDLE = 9212  # старое — Middle на VDS
```

на:

```python
VDS_MIDDLE_HOST = "127.0.0.1"   # твой ноут (если запущен локально)
VDS_MIDDLE = 9292
```

И в `peers={}` поменяй ключ Middle. После рестарта `murnet_proxy.py` circuit пойдёт **через твой ноут**.

## Когда **не** запускать ноут как узел

| Ситуация | Не запускай |
|---|---|
| Используешь публичную Wi-Fi | Светишь IP кафе/отеля |
| Лимитный мобильный интернет | Узел сжирает трафик |
| Корпоративная сеть на работе | IT-отдел не оценит |
| Ноут постоянно засыпает | Узел постоянно мерцает — это хуже чем его отсутствие |

## Юридический момент

В РФ есть закон 276-ФЗ об операторах анонимайзеров. Middle-узел технически попадает под него (теоретически), но прецедентов привлечения **только Middle**-операторов (не Exit) — нет. Tor middle-operators в РФ работают спокойно.

Если беспокоишься — поднимай узел не на ноуте, а на иностранном VPS (Hetzner DE €4/мес).

## Что делать когда выключаешь ноут

Просто `Ctrl+C` в окне с `home_relay.py`. Сеть автоматически перестроит circuits мимо тебя. На пользователях это не отразится (если есть другие узлы) — а пока узлов мало, твой shutdown = пользователи получат 502 → их клиент перестроит circuit через VDS-Middle.
